# 前端 UI 库引入需求

## 核心诉求

在保留 Writer 现有功能不变的前提下，引入一个 UI 组件库来提升前端视觉效果和交互体验。

## 现状分析

- 技术栈：Next.js 15.3 + React 19 + TypeScript
- 当前零 UI 库依赖，17 个手写组件
- 4000+ 行手写 CSS，基于 CSS 变量的完整设计系统
- 暖色调设计语言（teal/coral/gold），已支持 light/dark 主题
- 组件覆盖：Sidebar、ChatPanel、TracePanel、TokenChart、StyleModal、ConfirmDialog 等

## 已确认决策

### 1. "美观"的定义 → 组件质量提升
- 目标：更精致的 Modal、Dropdown、Tabs、Toast、Tooltip 等"标准 UI 组件"
- 不是推翻视觉风格，而是用高质量组件替换手写实现

### 2. 视觉调性 → 保留并增强
- 喜欢当前的暖色调（teal/coral/gold）羊皮纸质感
- 在此基础上做得更精致，不改调性

### 3. 改造优先级（从高到低）
1. **追踪面板（TracePanel）** — 最复杂、组件密度最高
2. **对话面板（ChatPanel）** — 消息列表 + 输入框
3. **整体布局** — Sidebar + TopBar

### 4. 主题定制能力 → 深度定制
- UI 库必须支持通过 CSS 变量 / Design Token 完全替换配色
- 新组件必须能完美融入现有暖色调体系（teal/coral/gold）
- 不接受 UI 库自带风格与现有设计的割裂

### 5. Toast / 通知机制 → 需要
- 当前错误提示用底部 `<p className="error-copy">` 展示，体验差
- 需要引入 Toast 通知系统（错误、成功、加载中等场景）
- 去掉底部的 error-copy

### 6. TracePanel 具体期望
- 用户说"说不上来，就是觉得粗糙"——缺乏具体锚点
- 需要由设计方案主动提出具体改进项供确认

### 7. 迁移策略 → 渐进替换
- 从 Toast（纯新增，零冲突）开始验证 UI 库
- 依次替换：Toast → TracePanel → ChatPanel → 整体布局
- 每步可验证、可回退，新旧组件允许短暂共存

### 8. ChatPanel 改进项 → 全部
- **消息气泡**：更精致圆角、阴影层次、出现/消失动画
- **工具状态指示器**：骨架屏/shimmer 加载效果、进度条、清晰状态图标
- **输入框**：拖拽调整大小、字符计数、快捷键提示、发送状态反馈

### 9. 响应式 → 本次不需要
- 只考虑桌面端，移动端适配后续再说

### 10. 整体布局改进 → 全部
- **Sidebar 导航项**：更精致的选中态、图标动画、可折叠分组
- **TopBar 工作区切换器**：更专业的 Select/Dropdown 组件、更好的已选状态展示
- **整体质感**：更柔和的分割、微阴影层次、面板切换过渡动画

### 11. UI 库偏好 → 无偏好
- 由方案推荐，约束条件：
  - 必须支持深度主题定制（CSS 变量 / Design Token）
  - 必须兼容 React 19 + Next.js 15
  - 无特别偏好或排斥的库

### 12. 暗色模式 → 能用就行
- 不要求新组件在暗色模式下与现有风格完美匹配
- 最低要求：不刺眼、不漏光（白底黑字）
- 暗色模式的精调可后续迭代
