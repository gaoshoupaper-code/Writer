# chapter-writing

正文章节的写作与修订执行流程。

## 上下文准备

写作前必须按以下顺序读取必要文件：

### 必读文件
1. `outline.md` — 了解故事整体结构，明确本章在全局中的位置。
2. `volume/volume-XX.md`（本章所在卷）— 了解卷内章节划分和节奏安排。
3. `volume/XX.md`（本章所在卷）— 了解卷内故事线和事件，明确本章涉及的故事线推进要求。
4. `character/*.md` — 了解出场角色的设定，确保人物行为符合设计。
5. `detail/chapter-XX.md`（对应本章编号）— 了解本章的规划约束。

### 前文上下文
- 写第 N 章时，读取第 `max(1, N-5)` 章到第 `N-1` 章作为前文上下文。
- 不足五章则读取已有全部前文。
- 前文由你自己去 `chapter/` 目录中读取，不要要求父代理提供。

## 细纲遵循规则

`detail/` 目录下的细纲文件是写作的约束框架：

- **关键 beat 是硬约束**：必须覆盖 `detail/chapter-XX.md` 中列出的所有 beat，不能遗漏或替换。
- **核心事件和情绪走向是强引导**：应尽量贴合，但允许写作过程中的自然波动。
- 如果 `detail/` 目录下不存在对应章节的细纲文件，则只依据 `outline.md` 和父代理提供的任务指令写作。

## 写作执行

根据父代理委托确定任务类型：

### 追加新章
- 将本章写入 `chapter/chapter-XX.md`。
- 保留章节标题结构。

### 修订指定章节
- 只替换指定的 `chapter/chapter-XX.md`，不要重写其他章节。
- 如果修订来自 review，必须覆盖原章节文件而不是另存为新文件。
- 优先解决 review 指出的关键问题。

## evolution 审查循环

完成章节写入后，进入 evolution 审查循环：

1. 调用 `evolution` 子代理审查章节质量。
2. evolution 会读取章节文件并写入审查报告到 `review/` 目录下对应文件，然后返回审查结论和修订建议。
3. 根据 evolution 返回结果决定下一步：
   - 返回 **"建议修订"** 或 **"必须修订"** → 读取 `review/` 下的审查报告，根据核心问题和修改建议修订同一个章节文件，然后**再次调用 evolution** 审查修订版本。
   - 返回 **"通过"** 或 **"无需修改"** → 直接向父代理返回结果。
4. 最多调用 evolution **3 次**（含首次审查），超过后系统会强制终止评估循环。
5. 返回父代理时，回复中包含：**修订轮数**、**是否有质量风险**、**是否需要调整大纲或状态**。
   - 格式示例：`修订轮数：1/3\n质量风险：无\n大纲调整：否`

## 注意事项

- 只修订同一个 `chapter/chapter-XX.md` 文件；不要新建章节，不要修改 `outline.md`、`character/`、`state_log.md` 或 `evaluation.md`。
- 如果确实需要调整大纲或状态，只在回复中标记建议，不要自行修改。
- evolution（review 子代理）每次审查都是独立的，不携带之前审查的历史。
- 你会积累上下文：在修订轮次中，你的历史对话会保留，让你能基于之前写过的版本和 review 反馈逐步改进。
