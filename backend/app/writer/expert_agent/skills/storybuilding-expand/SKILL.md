# storybuilding-expand

增量故事内容填充流程。

## 工作流程

### 步骤 1 — 前置检查

**检查 outline.md 是否存在：**
- 不存在 → 拒绝执行，返回：`错误：outline.md 不存在，请先使用初构 Skill 构建故事骨架。`
- 存在 → 继续执行

### 步骤 2 — 读取全局索引

**先读 outline.md 获取全局视图**：
- 锚点一览表：了解当前时间轴上有多少锚点
- 故事线全局表：了解所有故事线的类型、范围、状态
- 浮空内容清单：是否有需要重新连接的内容

### 步骤 3 — 按需读取目标文件

根据委托中的"本轮焦点"和"用户扩展方向"，读取需要的文件：
- 涉及某卷 → 读取 `volume/XX.md`
- 涉及人物 → 读取 `character/*.md`
- 涉及世界观 → 读取 `worldview.md`

**不要全量读取所有文件**——只读本次任务涉及的范围。

### 步骤 4 — 增量扩展

按照系统提示词中的内容规范和增量构建原则，扩展各维度内容。

#### 4.1 扩展操作

**可能的扩展操作：**
- 插入新锚点 → 主线自动拆分，形成新卷纲区间
- 丰富某卷的故事线段 → 添加事件、标注因果
- 添加新故事线 → 同步更新 outline.md 全局表
- 暗线浮出 → 更新类型标签
- 回收钩子 → 更新钩子追踪表
- 添加新人物 → 创建角色档案
- 深化世界观 → 读取后增量写入

#### 4.2 事件填充节奏

- 每次增量以 **3 个事件为基准单位**，允许 **2-4 个浮动**
- 事件间不要求形成因果链，可以并列

#### 4.3 故事线负载管控

| 故事线类型 | 阈值 | 超出动作 |
|-----------|------|---------|
| 主线 | > 5 事件 | 拆分：原线结束，新线以继承关系接替，事件按叙事段重新归类 |
| 次线 | > 3 事件 | 同上 |
| 暗线 / 角色线 | > 4 事件 | 标记警告，优先考虑浮出或收束 |

#### 4.4 锚点连接密度

- 两个锚点之间，**主线 + 支线 ≤ 3 条**（暗线和角色线不计入）
- 超过 3 条 → 在连接最密集的点附近插入新锚点

#### 4.5 人物引入规则

**触发条件（满足任一即可引入）：**
- **载体缺失**：新拆分出的故事线没有合适角色承载
- **角色过载**：某角色同时承载 > 4 条故事线
- **因果桥梁**：两个事件需要因果连接，但缺少中间执行者

**引入约束（必须同时满足）：**
1. 与 ≥ 1 个已有角色有关系（血缘/利益/恩怨/师徒……）
2. 服务于 ≥ 1 条已有故事线（不能自带一条全新独立线进来）
3. 引入事件与已有事件有时空或因果交集

#### 4.6 内容关联约束

所有新增内容不得完全独立，最低关联要求如下：

| 新增类型 | 最低关联要求 |
|----------|-------------|
| 新事件 | 属于 ≥ 1 条已有故事线，或与 ≥ 1 个已有事件有因果/时空交集 |
| 新故事线 | 共享 ≥ 1 个已有角色，或与 ≥ 1 条已有故事线有交叉事件 |
| 新人物 | 须同时满足 4.5 的三约束 |
| 新锚点 | 必须在已有时间轴上插入，不能创造平行时间线 |

#### 4.7 钩子推进约束

- 钩子埋设后，**≤ 2 个锚点内**必须有推进动作（推进/回收/废弃）

#### 4.8 卷密度管控

- 单卷总事件数 > 15 → 标记过载，下一轮优先扩展其他卷

#### 4.9 世界观同步

- 扩展中出现**新势力**（组织/门派/种族/国家）→ 同步写入 worldview.md
- 扩展中出现**新地点**（城市/区域/地标）→ 同步写入 worldview.md
- 写入最低信息量：名称、与已有势力的关系、在当前事件中的立场

#### 4.10 写入顺序与克制度

写入顺序（按需）：
1. `worldview.md`（先读取，按需深化）
2. `character/*.md`（先读取，增量深化或新增）
3. `outline.md`（先读取，更新锚点/全局表/详情）
4. `volume/XX.md`（先读取，扩展故事线/事件/钩子）

**克制度原则**：
- 只添加本次目标明确需要的内容，不做预防性扩展
- 已有内容不应被删除或大幅重写（除非评估发现一致性问题）
- 新增/变更故事线时，同步更新 outline.md 的故事线全局表
- **新事件级联同步**：引入新事件后，检查其对已有事件的影响（因果链变化、角色状态推移、钩子状态更新、时间线位移），对受影响的已有事件进行微调，确保一致性

### 步骤 5 — evolution 评估

完成所有产物写入后，调用 evolution 子代理进行评估。

**调用时在任务描述中附加**：
"本次为增量填充，请侧重评估内容饱和度、跨维度一致性和克制度（不臃肿）。已有内容不应被删除或大幅重写。"

**通过标准**：总分 ≥ 80 且无"必须修改"级别的核心问题 → 接受当前版本。
低于标准 → 读取 evaluation.md，修订对应产物，再次调用 evolution。

最多调用 evolution 3 次（含首次），超过后系统强制终止。

返回时包含：修订轮数、是否有质量风险。
格式：`修订轮数：1/3\n质量风险：无`

### 步骤 6 — 回复父代理

用纯文本简要汇报：
- 本轮扩展了哪些维度
- 新增了哪些具体内容（锚点/故事线/事件/人物）
- 当前全局状态（锚点数、故事线数、未回收钩子数）
- 评估状态
- 下一步建议

## 注意事项

- 只写入系统提示词中定义的文件路径，不修改其他文件。
- 增量扩展时先读取再修改，不要从零重写已有文件。
- 维度之间保持一致性：引用其他维度的内容时，确保被引用的内容确实存在。
- 操作前先读 outline.md 获取全局视图——这是最重要的前置步骤。
