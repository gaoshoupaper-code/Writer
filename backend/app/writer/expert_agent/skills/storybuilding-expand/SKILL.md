# storybuilding-expand

增量故事内容填充流程。每次扩展按**人物/故事线比值 R = 人物数 / 故事线数**分流为两种互斥模式：

- **模式A — 新增故事线**：R > 3（人物充足）时，新增一条完整故事线（自带三幕事件组）
- **模式B — 新增人物并融入**：R ≤ 3（人物不足）时，新增一个人物并融入现有故事，**不新增故事线**

每次调用只执行其中一种模式。

## 工作流程

### 步骤 1 — 前置检查

**检查 storyline.md 是否存在：**
- 不存在 → 拒绝执行，返回：`错误：storyline.md 不存在，请先使用初构 Skill 构建故事骨架。`
- 存在 → 继续执行

### 步骤 2 — 模式判定

**读取 storyline.md 获取全局视图**：
- 故事核心：了解故事定位和结局约束
- 故事线一览表：了解所有现有故事线的类型、状态

**按需读取其他文件**：
- 涉及已有故事线详情 → 读取 `storyline/S{XX}-{名}.md`（了解事件编排、定位交汇点）
- 涉及人物 → 读取 `character/*.md`
- 涉及世界观 → 读取 `worldview.md`

**统计当前规模并计算比值 R = C / S**：
- 人物数 C = `character/` 下已有角色档案数
- 故事线数 S = storyline.md 一览表中的故事线行数

| 比值 R | 模式 | 本轮动作 |
|--------|------|---------|
| R > 3（人物充足） | **模式A** | 新增一条故事线 |
| R ≤ 3（人物不足） | **模式B** | 新增一个人物并融入现有故事，**不新增故事线** |

判定后只走对应模式，两种模式互斥；以读取到的实际 C、S 为准，不要凭感觉。

### 步骤 3 — 深入思考该怎么加合适

模式判定只决定了"加故事线还是加人物"，但具体加什么、怎么加、加在哪还得先想清楚，避免机械套模板。按以下顺序思考：

1. **缺什么**
   本次委托焦点指向缺叙事维度（缺故事线）还是缺角色厚度（缺人物）？与步骤 2 的 R 值判定是否一致。

2. **观察已有内容的结构**
   先读懂现状再决定怎么加：
   - 事件紧密度：已有故事线的事件编排疏松还是密集？有没有可承接或可插入的缝隙？
   - 人物关系复杂度：已有角色间的关系网是单线还是交织？新内容该挂到哪条关系上？
   - 故事线走向：各线的起点、交汇点、走向，判断新内容与哪条线咬合最有张力。

3. **怎么加、加什么、加在哪**
   基于上一步的观察，确定具体加法：
   - 是承接某条已有故事线往下写（新线起点接旧线末尾），还是插入一条与已有线交汇的新线？
   - 新人物是前瞻登场（最新事件组之后追加）还是回溯嵌入（微调已有事件）？
   - 落到哪条线、与哪个角色建立因果、关联点（交汇事件 / 关系建立节点）具体在哪？

4. **按照规则去加**
   想清楚后，进入下方对应模式的详细规则执行（模式A / 模式B + 公共规则）。

## 模式A — 新增故事线

（当 R > 3 时执行）

### A.1 确定新增故事线类型

**约束：先判别本次新增的是什么线（支线 / 角色线 / 暗线），再编排内容——类型决定事件组模板与拓扑角色（主线串行、其余与主线并行），类型未定不得动笔。**

根据委托中的"本轮焦点"和"用户扩展方向"，确定本次新增的故事线类型：

| 类型 | 适用场景 |
|------|---------|
| 支线 | 扩展叙事维度，补充主线的世界切面 |
| 角色线 | 为已有核心人物和BOSS创建独立弧光 |
| 暗线 | 埋设深层信息，制造悬念 |

**类型选择约束**：
- 角色线：必须有对应的核心人物档案（角色类型为"核心人物"或"Boss"），每个核心人物最多 1 条角色线
- 暗线：适合需要隐藏信息逐步揭示的叙事需求
- 支线：最通用的扩展方式

### A.2 产出完整故事线

按系统提示词中的内容规范和事件组模板，新增一条完整故事线。

**产出内容**：
1. 新建 `storyline/S{XX}-{名}.md`，写入故事线详情（S{XX}，含全局走向）+ 完整事件组：
   - 主线：发展(8) + 终局(4)
   - 支线：发展(4) + 终局(2)
   - 角色线：弧光(5)
   - 暗线：悬念→冲突→危机→反转→揭露(5)
2. 在 storyline.md 的故事线一览表追加一行

**关联约束**：
- 新故事线必须与 ≥ 1 条已有故事线有交汇事件
- 交汇事件独立于事件组模板，作为额外插入的特殊事件；详情只写在本线文件（交汇时主属线=ID 最小者）
- 模式A 下若需附带引入新角色，仍须满足下方"人物引入约束"
- 故事线可续写：新增的故事线可承接上一条故事线的末尾往下写，新线起点与上一线末尾在时间与因果上连续衔接，不重复上一线已写过的内容

#### A.3 事件组插入规则

- 按模板固定序列生成，不可跳过、调换或拆分
- 每个事件独立拥有 E{XXX} ID 和完整字段
- 同组事件共享 G{XX} 组 ID
- 交汇事件不属于任何事件组，独立插入

## 模式B — 新增人物并融入

（当 R ≤ 3 时执行。本轮**不新增故事线**，只新增一个人物并融入现有故事）

### B.1 设计人物档案

按系统提示词的角色档案规范，新建 `character/{角色名}.md`，包含完整段落（基本信息、角色类型、叙事结构功能、外貌、性格、弧光、能力与局限、关系网络）。

**角色类型选择**：依据该人物在已有故事中要承担的功能，选 核心人物 / Boss / 配角（主角一般初构已定，不在此新增）。

### B.2 选定融入目标

融入不是把人物"贴"进故事，而是让他与已有结构产生因果。必须明确：

- **服务于哪条/哪几条已有故事线**（≥ 1 条）：新人物在哪条线上发挥作用
- **与哪个已有角色建立核心关系**（≥ 1 个）：关系类型（血缘/利益/恩怨/师徒/情感……）+ 动态变化方向

### B.3 融入方式（二选一或组合）

| 方式 | 做法 | 适用 |
|------|------|------|------|
| **前瞻登场** | 不改动任何已有事件；在目标故事线**最新事件组之后**追加一个新事件组或交汇事件，让新人物在此首次登场并承担叙事功能，其过往借"揭露/悬念"自然带出 | 常规引入新人物 |
| **回溯嵌入** | 选取 4-5 个已有事件，把新人物加入其"参与角色"，并**微调事件描述**以体现他的作用（如：他才是暗中激化冲突者、他提供了关键情报） | 伏笔回填——让读者感觉发现"这人一直在" |

**优先使用前瞻登场**；回溯嵌入须谨慎，且只微调、不推翻原事件核心。

### B.4 融入功能性要求（让融入"有用"而非贴牌）

- **承担明确功能**：新人物在所融入事件中必须是某种叙事功能的承担者——冲突制造者 / 关键信息提供者 / 力量变量（改变强弱对比）/ 情感锚点，不能只在"参与角色"列个名字
- **制造新张力**：新人物的加入要打破现有平衡，制造新的冲突、危机或悬念（呼应系统提示词的"戏剧张力/危机感"原则）
- **关系可演变**：与已有角色的关系要有"动态变化方向"（会发展、逆转或加深），不是静态标签

### B.5 融入落点（写入哪些文件）

| 文件 | 写入内容 |
|------|---------|
| `character/{新人物}.md` | 新建完整角色档案；"关系网络"含与 ≥ 1 已有角色的关系 |
| 目标故事线 `storyline/S{XX}-{名}.md` | 前瞻：追加新事件组/事件，新人物在"参与角色"；回溯：修订目标事件的"参与角色"和"描述" |
| 相关已有角色 `character/{已有角色}.md` | 在"关系网络"补与新人物的关系（双向一致） |
| `storyline.md` | **不新增一览表行**（本轮不新增故事线） |

## 公共规则（两种模式共用）

### 人物引入约束

模式B 必须全部满足；模式A 若附带新角色也须满足：

1. 与 ≥ 1 个已有角色有关系（血缘/利益/恩怨/师徒……）
2. 服务于 ≥ 1 条已有故事线（不能自带一条全新独立线进来）
3. 引入事件与已有事件有时空或因果交集

### 内容关联约束

所有新增内容不得完全独立：

| 新增类型 | 最低关联要求 |
|----------|-------------|
| 新故事线（模式A） | 共享 ≥ 1 个已有角色，或与 ≥ 1 条已有故事线有交汇事件 |
| 新人物（模式B） | 满足上方三条引入约束，并按 B.3 融入目标故事线 |
| 交汇事件 | 必须同时推动 ≥ 2 条故事线的方向改变 |

### 世界观同步

- 扩展中出现**新势力**（组织/门派/种族/国家）→ 同步写入 worldview.md
- 扩展中出现**新地点**（城市/区域/地标）→ 同步写入 worldview.md
- 写入最低信息量：名称、与已有势力的关系、在当前事件中的立场

### 写入顺序与克制度

写入顺序（按需）：
1. `worldview.md`（先读取，按需深化）
2. `character/*.md`（模式B 新建新人物档案 + 更新已有角色关系网络）
3. `storyline/S{XX}-{名}.md`（模式A 新建线；模式B 在已有线追加/修订事件）
4. `storyline.md`（仅模式A 在一览表追加一行；模式B 不动一览表）

**克制度原则**：
- 只添加本次目标明确需要的内容，不做预防性扩展
- 已有内容不应被删除或大幅重写（回溯嵌入仅限微调目标事件）
- 模式A 同步更新 storyline.md 一览表；模式B 不更新一览表

---

### 步骤 4 — evolution 评估（单次）

完成所有产物写入后，调用 evolution 子代理评估。**evolution 全流程只调用 1 次。**

**调用时在任务描述中附加**：
"本次为增量填充，请侧重评估新增内容与已有内容的一致性和克制度（不臃肿）。已有内容不应被删除或大幅重写。"

**单次评估策略**：
- 低于标准 → 读取 `evaluation.md`，修订对应产物**一次**，**不再二次评估**，直接进入返回步骤。

返回时包含：是否执行修订、是否有质量风险。
格式：`执行修订：否\n质量风险：无`

### 步骤 5 — 回复父代理

用纯文本简要汇报：
- 本轮执行的模式（A 新增故事线 / B 新增人物并融入）
- 模式A：新增了哪条故事线（类型、名称）、新增事件数、交汇事件情况
- 模式B：新增了哪个人物（角色类型）、融入到哪条故事线、采用前瞻/回溯、对该线做了什么
- 当前全局状态（故事线总数、人物总数、当前 R 值）
- 评估状态
- 下一步建议（若 R 仍 ≤ 3，建议下轮继续模式B；否则可走模式A）

## 注意事项

- 只写入系统提示词中定义的文件路径，不修改其他文件。
- 增量扩展时先读取再修改，不要从零重写已有文件。
- 维度之间保持一致性：引用其他维度的内容时，确保被引用的内容确实存在。
- 操作前先读 storyline.md 获取全局视图——这是最重要的前置步骤。
- 模式判定以读取到的实际 C、S 为准，不要凭感觉。
