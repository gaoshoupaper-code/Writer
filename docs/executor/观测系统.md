# 执行端观测系统详细文档

> 本文回答一个问题：**执行端（executor）的观测系统到底能观测到什么？怎么观测的？**
>
> 执行端是运行 AI agent 的后端（`executor/`）。它把 agent 执行的全过程记录成 **trace**，
> 这既是前端「链路追溯面板」的数据源，也是进化端（evolution）做评估/优化的数据源。
> 观测系统是平台层**最重的职责**，核心代码在 `executor/app/platform/trace/` + `platform/agent/middleware/trace_*.py`。

---

## 一、观测系统全景（三层结构）

```
┌────────────────────────── 产生层（executor 内）──────────────────────────┐
│                                                                            │
│   agent 执行                                                                │
│     │ 拦截 LLM/Tool 调用                     注册 run 层级父子关系            │
│     ▼                                        ▼                             │
│   TraceMiddleware（洋葱包裹模型/工具）   TraceCallbackHandler（LangChain 回调）│
│     │ 每次 start/end/error 产事件                                        │
│     ▼                                                                      │
│   TraceRecorder（核心记录器）                                               │
│     ├─ 增量存储（increment.py）：LLM input 只存新增尾部                     │
│     ├─ 投影（projector.py）：原始事件 → nodes/context/todos                │
│     ├─ 链摘要（chain_summary.py）：每节点一行可读描述                       │
│     ├─ 写盘解耦：deque 缓冲 + 后台 drain 协程成批落盘                       │
│     └─ 摘要导出（summary_export.py）：trace 结束自动生成 *_summary.json     │
│                                                                            │
│   落盘产物（每个 workspace 下）：                                            │
│     traces/index.json                          ← trace 索引（status/时间）  │
│     traces/20260702-1430/trace-xxx.jsonl       ← 原始事件流                 │
│     traces/20260702-1430/trace-xxx_summary.json ← 模型可读摘要              │
│                                                                            │
│     完成后 POST 通知 evolution 摄入（_notify_evolution）                     │
└────────────────────────────────┬───────────────────────────────────────────┘
                                 │
                    contracts/trace/（两端共享的单一真源 schema）
                                 │
┌────────────────────────── 消费层（evolution 内）─────────────────────────┐
│                                                                            │
│   ingestion/（摄入）         loader → importer → projector → DB            │
│   view/traces.py（trace 查询 API：列表/详情/删除）                            │
│   view/active.py（活跃大盘：轮询执行端内存中的活跃 trace）                    │
│   view/stats.py（宏观统计：概览/skill 排行/失败模式/时间线）                  │
│   common/flow_metrics.py（流程硬指标：拓扑/可靠性/资源消耗）                  │
│   eval_agent/tools/trace.py（评估 Agent 读 trace 的三件套工具）              │
│   frontend/components/trace/（链路时间线 + token 图表 + 抽屉详情）           │
└────────────────────────────────────────────────────────────────────────────┘
```

**关键认知**：
- 数据 schema 是**单一真源**——定义在仓库顶层 `contracts/trace/__init__.py`，执行端和进化端各自 re-export，两边永远一致。
- 执行端只管**产生 + 落盘 + 通知**；进化端管**摄入入库 + 查询展示 + 算指标**。
- 一条 trace = 一次 agent 执行的完整录像，从 `run_start` 到 `run_end/run_cancelled/run_error`。

## 二、能观测到什么（核心）

观测的数据按**节点维度（node kind）**组织。一条 trace 投影后会变成一棵节点树 + 一串上下文片段 + 一串 todo 快照。

### 节点树（7 种节点类型）

| 节点类型 | 观测对象 | 能看到的信息 |
|---------|---------|------------|
| **`run`** | 一次完整执行 | endpoint（哪个接口）、status、起止时间、总耗时、事件数、错误 |
| **`agent`** | 一个代理实例 | agent_name、角色（main/subagent）、调用深度、所属并行组、耗时占比 |
| **`llm`** | 一次模型调用 | model_name、input（完整输入）、output、token 用量（input/output/total）、耗时、调用的工具列表、并行组 |
| **`tool`** | 一次工具调用 | tool_name、tool_args（交付物类工具保留正文）、tool_output、耗时、status、并行组 |
| **`skill`** | 一次技能调用 | skill_name（从读 SKILL.md 的 read_file 事件自动识别）、耗时 |
| **`todo`** | 一次任务清单更新 | 所有 todo 项、状态（pending/in_progress/completed）、当前活跃项 |
| **`error`** | 一次错误 | 错误类型（llm_error/tool_error/run_error）、错误信息、发生位置 |

### 每个节点上能读到的具体字段

**LLM 节点**（最关键，是诊断 agent 行为的主要材料）：
```
node_id / parent_node_id（挂到哪个 agent）
kind = "llm"
agent_name + agent_role（main/subagent）+ depth
model_name（用了哪个模型）
started_at / ended_at / duration_ms
usage: { input_tokens, output_tokens, total_tokens }   ← token 成本
context_anchor_id        ← input 内容的锚点
input_context_range      ← input 增量引用范围
output_context_anchor_id ← output 内容的锚点
tool_calls               ← 本次模型请求的工具调用列表
chain_summary            ← 一行可读摘要（模型名 + 输出前 100 字）
parallel_group_id        ← 并行调用组（一次返回多个 tool_calls）
raw_event_ids            ← 对应的 llm_start/llm_end 原始事件 id
```

**Tool 节点**：
```
tool_name（如 write_file / read_file / task / ask_user ...）
tool_args（write_file/edit_file 的 content 完整保留，其余工具清空控体积）
tool_output（工具返回值）
duration_ms / status（failed 时有 error）
skill_name（若该 tool 是读 SKILL.md，会被识别成 skill 节点）
```

**Agent 节点**（实例化拆分——同一个子代理被多次委托会出现多个节点）：
```
agent_name（如 meta-agent / outline-subagent / writing-evaluation-subagent）
agent_role: main / subagent
depth: 0（main）/ 1（subagent）/ 2（evaluation 子代理）
parallel_group_id
started_at / ended_at（从首末事件时间戳回填 duration）
```

### 上下文片段（context segments）

每个 LLM/Tool 的输入输出都被切成**上下文片段**存起来，每段含：
```
anchor_id（稳定锚点）+ sequence（顺序）
kind: system / human / ai / tool / todo / error / skill
agent_name + agent_role + depth
title + content（实际内容）
metadata（phase/event_type/model_name/duration_ms/usage/tool_name...）
tool_call_names（本次模型想调的工具）
related_node_id（关联到哪个节点）
```

### Todo 快照

每次 `write_todos/write_todo` 工具调用会产生一个 todo 快照：
```
anchor_id + agent_name
items: [{ id, content, status }]   ← 全部任务项
active_item                          ← 当前 in_progress 的任务
```

## 三、观测到的 12 种事件类型

recorder 写进 jsonl 的原始事件，是观测数据的最低层。所有结构化节点都是从事件投影出来的。

| 事件类型 | 触发时机 | 关键负载 |
|---------|---------|---------|
| `run_start` | trace 创建 | endpoint / thread_id / workspace_id / **user_id**（归属键）/ **run_purpose**（user_generation vs optimization，防自指） |
| `run_end` | 正常完成 | duration_ms |
| `run_error` | agent 抛错 | error / duration_ms |
| `run_cancelled` | 被取消 | error（区分 user_stop / client_disconnect / timeout）/ duration_ms |
| `run_awaiting` | 进入 HITL 等待（ask_user） | status = awaiting_input（非终态） |
| `run_meta` | trace 收尾时写入元信息 | input.prompt_versions（本次用的 prompt 版本） |
| `llm_start` | 模型调用开始 | **完整 input**（system prompt + 注入上下文 + 对话历史，已截断）/ model_name / run_id / parent_run_id |
| `llm_end` | 模型调用完成 | output / **usage（token）** / tool_calls / duration_ms |
| `llm_error` | 模型调用出错 | error / duration_ms |
| `tool_start` | 工具调用开始 | tool_name / tool_call_id / run_id / parent_run_id |
| `tool_end` | 工具调用完成 | tool_output / duration_ms |
| `tool_error` | 工具调用出错 | error / duration_ms |

**事件通用字段**（每条事件都有）：
```
trace_id / event_id / sequence（单调递增）/ type / status / timestamp
source: "system" | "middleware"   ← 区分系统事件 vs 中间件拦截事件
duration_ms
run_id / parent_run_id / parent_event_id   ← 构建调用树
agent_name / node_name
output_anchor_id（稳定锚点，永久不变）
input_context_range（增量存储指针）
```

**特别说明——HITL 与取消的状态**：trace 有 5 种 status —— `running / awaiting_input / completed / failed / cancelled`。`awaiting_input` 是唯一非终态，表示 agent 在 `ask_user` 处暂停等人输入。

## 四、核心机制（怎么观测的）

### 1. 事件拦截：TraceMiddleware（洋葱模型）

`platform/agent/middleware/trace_middleware.py` 以 DeepAgents 中间件的形式包裹每次模型调用和工具调用：

```
wrap_model_call:  _record_model_start  →  handler(request)  →  _record_model_end / _record_model_error
wrap_tool_call:   _record_tool_start   →  handler(request)  →  _record_tool_end  / _record_tool_error
```

- 每次调用前后各产一条事件（start/end/error 三态）。
- **GraphInterrupt 不算错误**：ask_user 的 interrupt 是正常控制流，特意放行不记 llm_error/tool_error，避免监测面板误报。
- **token 用量提取**：兼容多供应商（input_tokens/prompt_tokens、output_tokens/completion_tokens 等不同字段名），统一归一化为 `{input_tokens, output_tokens, total_tokens}`。
- **单条事件体积控制**：`_truncate_payload` 把超长字符串截断到 10000 字符（保留头尾各半），因为 llm_start 会把完整 system prompt + 注入的全部上下文文件记进去，单条原本几十 KB。

### 2. 调用树构建：TraceCallbackHandler

`platform/agent/middleware/trace_callback.py` 是 LangChain 回调，在 `on_chat_model_start / on_chain_start / on_tool_start` 时把每次 run 的父子关系（run_id → parent_run_id）登记进 recorder。projector 用这些关系把扁平事件组织成调用树。

### 3. 增量存储（控体积的关键）

`platform/trace/increment.py`——同一个 trace 内，第 N 次 LLM 调用的 input 通常包含第 N-1 次的 input（历史累积）。只存「新增尾部消息」+ 一个指向前次范围的指针（`input_context_range`），重建时顺着 anchor 链回溯拼接。

- **去重 key 用结构身份**（type + content 前 200 字符指纹 + tool_call_id），不用内容哈希——因为 middleware 截断会让内容边界微妙变化，哈希不可靠。
- **降级安全**：recorder 跨重启丢失内存索引后，该 trace 退化为全量存储（range 为空），不报错。
- **重建端必须「先算后写」**：否则边算边把重建结果写回会让 collected_messages 指数膨胀撑爆内存（代码注释里有这个踩坑记录）。

### 4. 写盘解耦（不让观测拖垮主流程）

`recorder.py` 的 `append_event` 原来同步 open+write，单次生成写上千个事件会周期性占满事件循环，导致生成期间所有别的 HTTP 请求排队超时。修复方案：

- `append_event` 只把待写行 append 进 `_pending_writes`（deque，微秒级纯内存）。
- 后台 `_drain_loop` 协程每 0.5s 成批落盘（`asyncio.to_thread`，不占事件循环）。
- trace 结束（complete/fail/cancel）时同步 flush 该 trace 残余事件保证完整。
- 写盘失败静默吞掉——trace 是派生数据，不可拖垮主流程。

### 5. 僵尸清理（awaiting_input 超时回收）

`awaiting_input` 超 2h 未 resume 的 trace 会被后台扫描任务标成 `cancelled`（reason=timeout）。扫描间隔 5min。三种取消来源用 `error` 字段区分：
- `user_stop`：用户点了停止按钮
- `client_disconnect`：SSE 连接断开（cloudflared/Cloudflare 超时、网络抖动）
- `timeout`：awaiting_input 超 2h 未响应

### 6. 摘要导出（喂给 evolution 的模型可读格式）

`summary_export.py`——每条 trace 结束后自动生成 `trace_xxx_summary.json`，节点级粒度，每个节点只保留关键信息（几十 token），通过 `node_id` / `raw_event_ids` 可回查原始 jsonl。这是 evolution 的 judge/optimizer 高效消费的入口。

### 7. 链路摘要（chain_summary）

`chain_summary.py` 为每个节点生成一行人类可读描述，让 trace 列表/时间线一眼能看懂：
- tool 节点：`write_file: 路径` / `read_file: 路径` / `set_goal: 目标`
- llm 节点：`模型名: 输出前 100 字`（或列工具名）
- error 节点：`❌ 错误信息`
- skill 节点：`📖 技能名`

## 五、观测数据的消费（谁在用）

### 1. 进化端摄入（ingestion/）

执行端 trace 完成后 POST 通知 evolution（`_notify_evolution`，失败静默降级，靠 evolution 兜底扫描补漏）。evolution 把 jsonl 摄入自己的 SQLite：runs 表 + nodes 表 + event_payloads 表。摄入时重新投影 + 重建增量 input，保证 nodes/context/todos 一致。

### 2. 评估 Agent 的 trace 读取工具（三件套）

`evolution/app/eval_agent/tools/trace.py` 给评估子代理三个工具，按粒度从粗到细：
- **`read_trace(trace_id)`** —— 摘要层：所有节点结构化摘要，先纵观全局定方向。
- **`read_trace_node(node_id)`** —— 细节层：单节点完整 context（按 anchor 回溯）。
- **`read_trace_range(start, end)`** —— 区间层：连续 anchor 区间的完整对话流。

### 3. 流程硬指标（flow_metrics）

`evolution/app/common/flow_metrics.py` 从 trace 算三类**确定性指标**（无 LLM 成本），交评估子代理判断好坏：

| 类别 | 指标 |
|------|------|
| **协作拓扑** | subagent 调用次数/顺序、委托链深度、并行组数、各 subagent 耗时占比、evaluation 子代理数 |
| **错误保障** | tool 错误率、错误按类型/agent 分布、middleware 拦截事件数、HITL 等待次数、review 调用数 |
| **资源消耗** | 总 token、input/output token、各 agent token 占比、平均每次 LLM token、重复读同一文件次数及浪费量 |

### 4. 宏观统计面板（stats）

`evolution/app/view/stats.py` 提供 4 类聚合指标（实时 GROUP BY）：
- **overview**：总量、成功/失败、错误率、延迟分位（p50/p90/p99）、token 总量
- **skills**：agent 调用排行（调用数、平均耗时、失败率）
- **timeline**：trace 数量随时间变化（错误率趋势，按时间桶）
- **failures**：失败模式 TopN（按 error 文本前缀聚类）

### 5. 活跃大盘（active）

`evolution/app/view/active.py` 定期轮询执行端 `/internal/active-runs`（即 `recorder.list_active_runs`），展示**正在运行的 trace**——纯内存、不入库，含 endpoint、已运行时长、事件数、状态（running/awaiting_input）。

### 6. 前端追溯面板（components/trace/）

- **TraceChainTimeline**：节点树时间线（agent/llm/tool/skill 嵌套，并行组可视化）
- **TokenChartPanel**：token 消耗图表（按 agent / 按阶段分布）
- **TraceChainDrawer**：单节点详情抽屉（点 LLM 节点看完整 input/output 消息）
- **TracePanel**：列表 + 详情合一，支持 SSE 实时同步（运行中的 trace 实时刷新）

## 六、观测到的运行状态（5 种 status）

| status | 含义 | 进入条件 | 是否终态 |
|--------|------|---------|---------|
| `running` | 正在执行 | trace 创建 | 否 |
| `awaiting_input` | 等待用户输入（HITL） | agent 调 ask_user → interrupt | 否（可 resume） |
| `completed` | 正常完成 | run_end | 是 |
| `failed` | 出错 | run_error（agent 抛错） | 是 |
| `cancelled` | 被取消 | run_cancelled（user_stop/client_disconnect/timeout） | 是 |

`run_purpose` 字段区分「用户生成」(user_generation) 和「优化执行」(optimization)——后期优化闭环只处理前者，防自指断路。

## 八、一句话总结

> 执行端观测系统 = **TraceMiddleware 拦截每一次 LLM/Tool 调用** → **TraceRecorder 增量落盘成 jsonl + 自动导出模型可读摘要** → **TraceProjector 投影成节点树/上下文/任务快照** → **通知 evolution 摄入** → 供「前端追溯 / 评估诊断 / 流程指标 / 宏观统计 / 活跃大盘」五路消费。
>
> 能观测到：**每一次模型调用（含完整输入输出和 token 成本）、每一次工具调用、每一次技能调用、任务清单的每次更新、所有错误、HITL 等待、并行调用结构、子代理委托链**——基本是 agent 执行的完整可重放录像。
