# 细纲结构规范

本文档定义 Writer 系统中细纲的核心信息结构。每个线程工作目录下一个 `detail/` 目录，由 outline.md 的分段向下拆解生成，为 writing 子代理提供逐章写作指引。

## 定位

细纲是大纲与正文之间的桥梁。大纲回答"故事整体怎么走"，正文回答"每一句怎么写"，细纲回答"每一章具体写什么"。

```
outline.md（分段级：3-5 段）
    ↓ 拆解
detail/（章节级：每段拆为若干章）
    ↓ 指引
chapter/chapter-XX.md（正文：每章约 1000 字）
```

细纲不替代写作创造力——它约束"必须达成什么"，不规定"怎么写"。

## 为什么拆分成目录

单文件方案（`detail_outline.md`）在长篇场景下有两个问题：

1. **writing 子代理上下文浪费**：写第 15 章时只需要第 15 章的计划，但必须读取包含所有章节的单文件。
2. **更新成本高**：调整第 8 章的 beat 需要重写整个文件，出错的概率随文件变长而上升。

拆分成 `detail/` 目录后，writing 子代理只读当前章的文件 + overview.md，meta agent 更新时只改对应的章节文件。

---

## 目录结构

```text
detail/
├── overview.md              章节规划总览 + 线索调度
├── chapter-01.md            第 1 章细纲
├── chapter-02.md            第 2 章细纲
├── ...
└── chapter-XX.md            第 XX 章细纲
```

遵循项目已有的目录模式（`chapter/`、`review/`、`character/`），每章一个独立 Markdown 文件。

## overview.md — 章节规划总览与线索调度

回答"整本书的章节如何分布、节奏如何安排、线索如何调度"。为 meta agent 提供全局调度视角。

### 字段说明

**规划总览：**

| 字段          | 说明                                                                         |
| ------------- | ---------------------------------------------------------------------------- |
| 章节总数      | 计划写多少章                                                                 |
| 分段-章节映射 | outline.md 主线每一段对应哪几章，用区间表示（如"第一段：第1-3章"）           |
| 节奏设计      | 整体节奏规划：哪些章快、哪些章慢，以及快慢交替的理由                         |

**线索调度：**

每条线索记录：

| 字段         | 说明                                           |
| ------------ | ---------------------------------------------- |
| 线索名称     | 对应 outline.md 中的主线或次线名称             |
| 活跃章节     | 这条线索在哪几章有推进（列出章节编号）         |
| 关键节点章节 | 这条线索的核心转折或高潮发生在哪一章           |

每个交织点记录：

| 字段       | 说明                                           |
| ---------- | ---------------------------------------------- |
| 交织描述   | 引用 outline.md 中的交织点描述                 |
| 对应章节   | 这个交织点安排在哪一章发生                     |
| 产生的张力 | 在这一章中交汇催生了什么具体的冲突或转折       |

线索调度不是简单的时间表——它确保每条次线有足够的篇幅推进，交织点出现在最有效的位置，避免次线长时间消失或交织点堆叠过密。

### Markdown 模板

```markdown
# {作品标题} — 章节规划总览

## 规划总览

章节总数：
节奏设计：

分段-章节映射：
- 第一段：{分段标题} → 第X-X章
- 第二段：{分段标题} → 第X-X章

## 线索调度

### 主线：{主线名称}

活跃章节：第X、X、X章
关键节点章节：第X章

### 次线：{次线名称}

活跃章节：第X、X章
关键节点章节：第X章

### 交织点

- **{交织描述}**：对应第X章。产生的张力：{交汇催生了什么}。
```

## chapter-XX.md — 逐章规划

回答"这一章必须达成什么"。每章一个文件，作为 writing 子代理的章节任务来源。

### 字段说明

| 字段     | 说明                                                                       |
| -------- | -------------------------------------------------------------------------- |
| 章节标题 | 简短概括本章核心（2-6 字），兼顾暗示性和辨识度                            |
| 所属分段 | 对应 outline.md 主线的哪一段                                               |
| 本章目标 | 这一章在叙事上必须达成什么（一句话：推进了什么、改变了什么、释放了什么）   |
| 核心事件 | 这一章发生的核心因果事件（因为……所以……），聚焦动作和结果，不写氛围描写     |
| 出场人物 | 本章出场的角色，引用 `character/` 目录下的角色姓名                         |
| 关键 beat | 本章必须发生的 2-4 个叙事节点（按顺序），每个 beat 用一句话概括            |
| 情绪走向 | 本章的情绪弧线：入场情绪 → 出场情绪（如"平静 → 紧张"，"希望 → 失望"）    |
| 衔接要点 | 与前后章的承接关系：上一章留下了什么悬念或状态，本章结尾要为下一章铺垫什么 |

这些字段直接对应当前 meta agent 调用 writing 子代理时需要提供的参数（章节编号、剧情大纲、本章目标、出场人物、必须发生的 beat、承接关系），让调度从 ad-hoc 构造变为结构化读取。

**设计约束：**

- 关键 beat 是硬约束：writing 子代理必须覆盖所有列出的 beat，不能遗漏或替换。
- 情绪走向是软约束：引导基调但不强制每一段都贴合，允许写作过程中的自然波动。
- 核心事件聚焦因果：不是"角色做了什么"的罗列，而是"因为什么、导致了什么"的因果链。
- 出场人物控制密度：一般 2-4 人，避免单章出场过多导致分散。

### Markdown 模板

```markdown
# 第X章：{章节标题}

所属分段：
本章目标：
核心事件：
出场人物：
关键 beat：
  1.
  2.
  3.
情绪走向：{入场情绪} → {出场情绪}
衔接要点：承接上一章的{什么}；为下一章铺垫{什么}。
```

## 各角色读取策略

拆分后，不同角色按需读取：

| 角色           | 读取内容                                          | 原因                       |
| -------------- | ------------------------------------------------- | -------------------------- |
| outline 子代理 | `detail/overview.md` + 全部 `detail/chapter-*.md` | 生成/修订时需要全局视角    |
| writing 子代理 | `detail/overview.md` + 当前章的 `detail/chapter-XX.md` | 只需要当前章的计划和全局上下文 |
| review 子代理  | 当前章的 `detail/chapter-XX.md`                   | 对照细纲验证是否达成目标   |
| meta agent     | `detail/overview.md`                              | 调度决策需要全局节奏视角   |

writing 子代理是最高频的调用方，拆分后它的上下文从"所有章节"缩减到"2 个文件"，显著降低 token 消耗和干扰。

## 与现有系统的关系

### 向上：依赖 outline.md

- 章节通过"所属分段"字段向上追溯到 outline.md 的主线分段。
- 出场人物引用 `character/` 目录下的角色文件。
- 线索调度与 outline.md 的主线、次线、交织点一一对应。

### 向下：驱动 chapter/ 写作

- meta agent 调用 writing 子代理时，从 `detail/chapter-XX.md` 读取对应章节的完整任务参数，无需 ad-hoc 构造。
- writing 子代理的关键 beat、本章目标、衔接要点直接来自细纲字段。
- review 子代理可以对照 `detail/chapter-XX.md` 验证章节是否达成了预定目标。

### 更新时机

- **初次生成**：outline.md 通过 evaluation 后，由 outline 子代理生成 `detail/` 目录下的全部文件。
- **大纲变更**：outline.md 的结构骨架或主线分段变化时，先更新 `detail/overview.md` 的映射，再更新受影响章节的 `detail/chapter-XX.md`。
- **动态调整**：写作过程中如需增减章节或调整节奏，只改对应的章节文件和 overview.md 的映射部分。

## 设计原则

1. **按章拆分，按需读取**：writing 子代理只读当前章 + overview，不浪费上下文。
2. **向下追溯，不向上发散**：细纲的每一章都能追溯到 outline.md 的分段，不引入大纲中没有的全新故事方向。
3. **约束必须项，放开写法**：关键 beat 和核心事件是硬约束，情绪走向和衔接要点是软约束——writing 子代理在达成硬约束的前提下有充分的创作自由。
4. **一章一聚焦**：每章只做一件事（一个核心事件、一个情绪弧线），不贪多。如果一章需要做两件以上重要的事，拆成两章。
5. **线索不遗忘**：overview.md 的线索调度确保每条次线在合理的间隔内有推进，交织点出现在预定的章节位置。
6. **灵活适配**：短篇可以只有 3-5 章不设次线；长篇可以 20+ 章配多条次线和密集交织点。维度是约束不是枷锁。
7. **与角色档案联动**：出场人物引用 `character/` 目录下的角色姓名；角色的核心欲望和关系动态驱动章节中的冲突和 beat 设计。
