# evolution 文件大地图

> **怎么看这张图**：evolution 是 executor 的"教练系统"——观察 executor 运行、诊断问题、优化提示词。
> Phase 2 重构后，它的代码按**流水线阶段 + 公共底座**物理分层组织（不再是平铺）。
> 每张卡片先告诉你"这层管什么"，再逐个列出文件和它们的作用。

## 物理分层总览

```
evolution/app/
├── main.py              ← 启动入口（唯一留在 app/ 根的文件）
├── core/                ← 公共底座（db/llm/settings/models，所有层共用）
├── ingestion/           ← 采集层（trace 摄入 → 投影 → 入库）
├── diagnosis/           ← 诊断层（规则标红 → LLM 评估 → 规律挖掘）
├── improvement/         ← 改进层（优化 → 验证 → 实验编排）
└── view/                ← 查看层（统计/详情/可视化看板）
```

**分层铁律**：core/ 不得依赖任何业务层（由 `check_layering.py` 的 RE1 规则机器校验）。

## 🏛️ core/ — 公共底座

> **这层管什么**：所有业务层共用的基础设施——数据库、配置、AI 调用、数据格式定义。
> 这层**绝对不能依赖业务层**（铁律 RE1），否则会造成循环依赖。

| 文件 | 行数 | 作用 |
|------|------|------|
| `core/settings.py` | 70 | 服务配置中心。集中定义 evolution 需要的所有配置项（数据库路径、轮询间隔、LLM 参数等），统一从环境变量读取。所有模块通过它拿配置，避免配置散落各处。 |
| `core/db.py` | 1076 | SQLite 数据层。管数据库连接、建表、迁移。evolution 所有数据（trace、规则、提示词版本、实验记录、用户映射缓存）都存这里，是唯一的持久化层。 |
| `core/llm.py` | 47 | LLM 调用封装。evolution 自己也要用 AI（judge 做评估、optimizer 优化提示词），所有 AI 调用都走这个统一封装。 |
| `core/models.py` | 49 | trace 数据结构（**已迁移至 contracts/trace/**）。本文件是过渡兼容入口，从共享契约层 `contracts.trace` re-export。 |
| `core/sso_auth.py` | 155 | SSO 鉴权中间件。桌面端登录 executor → cookie 传给 evolution → 本中间件回调 executor `/api/auth/me` 验证 session + user_id 白名单。**区分 503/401**：executor 不可达（网络超时/宕机）返回 503（前端静默重试、不跳登录），session 真失效才返回 401（前端跳登录）。进程内 60s TTL 缓存减少回调。 |

## 📥 ingestion/ — 采集层

> **这层管什么**：把 executor 跑完留下的 trace（执行轨迹）收进来、读出来、转成结构化数据、存进数据库。
> 这组是 evolution 的"进货渠道"——没有它，后面所有诊断和改进都无米下锅。

| 文件 | 行数 | 作用 |
|------|------|------|
| `ingestion/ingestion.py` | 210 | trace 摄入路由。提供 `POST /ingestion/notify` 接口，executor 每跑完一次就调这个接口通知 evolution"有新 trace 了"。收到通知后调 executor 的 `GET /internal/traces/{trace_id}` 拉取 trace 内容（Phase 3 重构后不再读文件系统）。还提供 `GET /ingestion/active-key`：executor 的 build_writer_model 从此拉取激活 LLM 配置明文（api_key/base_url/model），打通桌面端配置 → executor 执行。两者都挂 /api/ingestion/ 前缀，受 X-Notify-Token 内网鉴权。它是 executor → evolution 数据流的**正式入口**。 |
| `ingestion/scan.py` | 95 | 兜底扫描。定时调 executor 的 `GET /internal/traces` 列表端点，补收那些"executor 忘了通知"的 trace（Phase 3 重构后不再扫文件系统）。它是 ingestion 的保险，防止因为通知失败而漏数据。 |
| `ingestion/loader.py` | 80 | jsonl 读取器。trace 的原始格式是 jsonl（每行一个事件的文本文件），这个模块负责把它解析成内存里的事件列表，供下游处理。 |
| `ingestion/importer.py` | 158 | trace 摄入器（串联者）。把 loader 读出的事件，经过 projector 投影，写入 SQLite 三张表。它是"读 → 投影 → 存"这条流水线的串联者。 |
| `ingestion/projector.py` | 900 | **trace 投影引擎（全域最大文件）**。把原始的事件流转换、归纳成结构化的数据模型——每个 agent 节点、运行摘要、token 用量等。它是 evolution 数据处理的**核心枢纽**。 |
| `ingestion/chain_summary.py` | 177 | 链路摘要生成。为每个 trace 节点生成一行简短的描述，让结构化数据变得**可读**。 |
| `ingestion/user_sync.py` | 80 | 用户名映射同步。定时调 executor 的 `GET /internal/users` 拉取用户列表，全量覆盖写入本地 `user_cache` 表（user_id→username 映射）。供 trace 历史列表展示用户名（evolution 不维护用户主数据，只缓存映射）。 |

## 🔬 trace/ — 进化端自观测层

> **这层管什么**：进化端自己的 trace 记录系统（从执行端 recorder 移植改造）。
> 评估 Agent 和进化 Agent 运行时，这层负责拦截 LLM/Tool 调用、记录事件流、
> 供 SSE 实时推送和终态回放。与 ingestion/（摄入执行端 trace）是两条独立链路。

| 文件 | 行数 | 作用 |
|------|------|------|
| `trace/recorder.py` | 892 | 进化端 trace 记录器（单例）。事件产生 → 内存缓冲（deque+Queue）→ drain 批量落盘（DB event_payloads + jsonl WAL）。终态批量投影 nodes。Phase 2 新增运行期投影 diff 能力：`project_and_diff`（全量投影+diff 出 node patch，供 SSE 推送）、`project_full_nodes`（终态全量 snapshot）、`_load_raw_events`。`cancel_run` 支持 user_stop reason（手动停止进化/评估 Agent 时收尾用）。 |
| `trace/differ.py` | 87 | **投影 diff 引擎（Phase 2 新增）**。`NodeSnapshotDiffer` 维护单 trace 的 nodes 快照，`diff()` 产出增量 patch（appended/updated），单次 O(N) 比对。让 SSE 只推 node 级 diff 而非全量重算，根除前端 O(N²)。 |
| `trace/increment.py` | 285 | LLM input 增量计算（write 时算）。`compute_increment` 用最长公共前缀切出新增尾部，控空间。`IncrementState` 维护增量状态（重启后丢失→降级全量）。 |
| `trace/trace_middleware.py` | 595 | LangChain 拦截中间件。拦截 LLM/Tool 调用产生 trace 事件（llm_start/end、tool_start/end/error），截断大字符串（单条 1 万字符上限）。 |
| `trace/trace_callback.py` | 74 | LangChain callback handler。注册 run 父子关系（内存，不持久化）。 |
| `trace/summary_export.py` | 81 | trace summary JSON 导出（模型可读格式，供 judge 评估用）。 |

|------|------|------|
| `diagnosis/rules_engine.py` | 93 | 阈值型规则引擎。trace 入库后立刻跑一遍规则，命中阈值就给 trace 打标。纯数值判断，不调 AI，最快。 |
| `diagnosis/rules.py` | 214 | 规则管理路由。提供规则的增删改查 API，还能查"某条 trace 命中了哪些规则"。 |
| `diagnosis/judge.py` | 273 | LLM-judge 评估引擎。对被判异常的 trace 做**深度质量评估**——规则引擎只能判数值，judge 用 AI 评判"这次运行到底好不好、问题在哪"。 |
| `diagnosis/rubric.py` | 78 | judge 的内置评分标准。定义"什么算好、什么算坏"的默认标准（rubric，评分细则）。 |
| `diagnosis/rubrics/xianxia.py` | — | 仙侠题材专属评分规则。针对特定创作题材的细化评分标准。 |
| `diagnosis/calibrate.py` | 240 | judge 方差校准。LLM 打分有随机性，这个模块校准 judge 的稳定性，让评分结果可信。 |
| `diagnosis/diagnosis.py` | 167 | badcase 诊断器。对失败案例做专项诊断，定位"为什么失败"。 |
| `diagnosis/mining.py` | 342 | Mining 引擎。从大量失败 trace 里**挖掘规律**。结合结构化预筛和 LLM 提炼，把零散坏案例聚成有意义的模式。 |
| `diagnosis/eval_extractor.py` | 202 | 评估输入提取器。按子代理类型，从 trace 里提取出"该被评估的交付物"。judge 要评估什么，由它先抽出来。 |

## 📦 harnesses/current/ — Agent 包（Phase 7 包化核心产物）

> **这组管什么**：一个**自包含的 Agent 包**——把 writing 流水线的全部"agent 定义"打成一个 Python package。
> executor 通过 `load_current_package()` import 它，调 `assemble(ctx)` 一行装配完整 agent。
> 这是 Phase 7 harness 包化重构的产物：把原本散在 executor 内的 prompt/subagent/middleware/skills/reviewer
> 物理迁移进这个目录，让 agent 定义可版本化、可进化、可 A/B 回放。
>
> **关键设计**：
> - `__init__.py:assemble(ctx)` 是唯一装配入口。接收 RuntimeContext（executor 构建，含 model/backend/styles 等），包内只读 ctx 不依赖 executor 其他状态。
> - 运行时值（model/backend/checkpointer/trace/workspace/styles）不进包，装配时由 executor 注入。
> - **M3 风格注入闭环后**：assemble 消费 `ctx.styles`，把风格 suffix 注入各 subagent 的 prompt（via apply_style_suffix）。
> - 包可被 evolution 迭代（改 prompt/middleware → 发版 → tar 快照存 DB）。

### 装配入口

| 文件 | 行数 | 作用 |
|------|------|------|
| `harnesses/current/__init__.py` | 170 | **assemble(ctx) — Agent 包装配入口**。读 prompts → 实例化 middleware → 组装 5 个 subagent（general-purpose + interview + storybuilding + detail_outline + writing）→ 消费 ctx.styles 注入风格 suffix → 调 create_deep_agent 返回编译后的 agent。是 executor 与包的唯一交互点。 |
| `harnesses/current/manifest.json` | — | 包元数据（version / parent_version / schema_lock / change_summary）。非装配指令——装配逻辑全在 assemble()。 |
